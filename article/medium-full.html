A Loop Is Only as Good as Its Gate

I broke an autonomous AI agent six ways to find out what actually catches failure.

A few months ago I let an AI agent improve itself overnight. It ran experiments, kept the wins, rolled back the losses, and left me a report with my coffee. The thing that made that safe wasn’t the agent. It was one number — val_bpb — that could fail a change without me in the room. Commit if it improved. Roll back if it didn’t. The gate, not the genius, is what let me close my laptop.

For two years the way you got work out of an AI agent was to prompt it by hand — type, wait, read the diff, type again. That era is closing. The leverage moved one floor up, to the loop: a small system that finds the work, hands it to the agent, checks the result, and decides the next move on its own. Everyone’s posting about it. Almost nobody is stress-testing the part that actually matters.

A loop is only as good as its gate. Take away the thing that can fail the work, and you don’t have an autonomous loop. You have an agent grading its own homework, on repeat, while the bill runs. There’s a name for it now — the loop that says “done” early and fails quietly.

And the dashboards you trust won’t save you — the traces, spans, and latency charts everyone stares at during an incident. The honest version: in the agent era, the chart that looks fine is often the one lying to you. Not because it’s broken. Because it answers “is it running?” when the question that matters is “is it right?”

So I put the gate on trial. I built an autonomous agent loop properly, then spent a week trying to break the one layer that’s supposed to keep it safe — and wrote down exactly what each layer of the stack could, and couldn’t, see.

Of six failure modes, the infrastructure telemetry — the stuff most teams page on — caught one. The trust layer caught all six. Including the one where the dashboards showed improvement while the quality quietly died.

What I Built

The agent is a production-grade AI-platform engineering assistant — a LangGraph agent answering ML training / inference / GPU-infra questions for an internal platform team, grounded in a real engineering corpus. The kind of autonomous agent a frontier org actually runs. Not a toy.

Engineer question
  → [INTAKE]     topic routing (training / inference / infra)
  → [RETRIEVER]  dense embedding retrieval over a real engineering corpus
  → [TOOLS]      real execution (sandboxed code, corpus search)
  → [RESPONDER]  grounded answer
  → [PROTECT]    Galileo quality gate   ← the loop’s gate
  → answer

Three layers, and the question each one answers:

  • BUILD — LangGraph: did we construct it correctly?
  • RUN — fleet telemetry (ClawTrace-style): is it running?
  • TRUST — Galileo: is it right? — the gate that makes the loop safe.

I called this the Trinity Stack. Then I broke it — six ways — to see which layer is actually load-bearing when the agent runs unattended.

The Metrics Running on Every Query

Every trace gets scored by Galileo’s server-side scorers, running on 100% of traffic — not a 5% sample:

  • context_adherence — are the answer’s claims grounded in the retrieved documents? (Galileo’s core RAG metric)
  • completeness — does the response fully address the question?
  • cites_kb_source — does the answer cite a specific knowledge source?

These are computed by Galileo’s platform, not by glue code in my app. (In this run the judge model was GPT-4o-mini via a configured integration; Galileo’s Luna-2 distilled judges are the option that makes 100%-of-traffic scoring dramatically cheaper than a frontier model — the economics that make “score everything” realistic.)

Baseline (correct configuration), Galileo-computed:
context_adherence 1.00 · completeness ~1.00 · cites_kb_source 1.00

Every number below is Galileo’s own — pulled from the Console after the platform scored the traces. No hand-computed metrics. Now let’s break things.

Failure 1: The Agent Dies (XL-1)

What I did: killed the process and deleted the heartbeat file.

  • Fleet: 🚨 alarm — heartbeat missing → page on-call.
  • Galileo: silence — traces simply stop. No new traces means nothing to evaluate.

The trap engineers fall into: “My traces stopped showing up in Galileo. Is the SDK broken?” No. The silence is accurate. If nothing is running, nothing gets logged.

Triage rule: traces stop suddenly + fleet alarm = RUN layer. Check your process before your SDK.

Recovery: restart the process → heartbeat resumes → traces resume. Zero Galileo action required.

Failure 2: The One That Keeps Engineers Up at Night (XL-2)

This is the most expensive failure mode in AI systems. Not the most dramatic — the most expensive.

What I did: swapped the engineering knowledge corpus for completely off-domain documents — cooking, yoga, plumbing, gardening, pet grooming. Same agent, same loop; only the index changed.

What the fleet layer saw throughout: process alive, heartbeat fresh, latency 679–1440ms (indistinguishable from baseline), error rate 0%. Every infrastructure health check: green.

What an engineer got for “How do I fix a CUDA out-of-memory error during training?”:

“To resolve this, shut off the water supply valve, remove the handle, and replace the washer and O-ring before re-tightening…”

That’s a plumbing repair. Professionally formatted. Completely wrong. Confidently delivered until someone noticed.

What Galileo saw (server-side scorers, baseline → poisoned):

completeness 1.00 → 0.00 🚨  ·  cites_kb_source 1.00 → 0.00 🚨  ·  context_adherence 1.00 → 0.60 🚨

The sharpest signals are completeness → 0.00 and cites_kb_source → 0.00: not one answer is useful, not one cites a real engineering source. The model just fabricates from parametric knowledge. Note the nuance — context_adherence only falls to 0.60, because some answers are technically “grounded” in the (wrong) retrieved docs. That’s exactly why you want more than one quality metric. Restore the corpus and all three snap back to 1.00.

Without Galileo, you find this out when an engineer follows a plumbing tip to fix a GPU OOM and files a ticket — after the agent has served wrong answers all day. And Galileo Insights clusters the failures by cause (“retrieval domain mismatch — off-domain corpus”) and points at the retriever: one insight, not a trace-by-trace audit.

This failure mode — healthy process, wrong corpus, confidently wrong answers — is fundamentally undetectable by fleet/infrastructure monitoring. Only semantic quality evaluation catches it. This is what Galileo was built for.

Failure 3: The Router Bug With a Visual Fingerprint (XL-3)

What I did: broke the LangGraph router — one line change that returns the same intent for every query regardless of content. Training questions, inference questions, infra questions all get routed to one flow.

Fleet: ✅ healthy. Galileo showed something elegant: in the Console trace view, every single trace followed the identical node path and retrieved the same documents. Normal routing has variety. A routing bug creates uniformity. That visual pattern is the fingerprint.

Galileo’s quality metrics stayed high (context_adherence and completeness ≈ 1.0) — the corpus is small enough that even wrong-category docs often contain something usable, so quality scores alone don’t catch this. A custom routing_accuracy scorer is registered to quantify the misroute. But the decisive signal here isn’t a number — it’s the trace view: every span path collapses to the same route.

The BUILD-via-TRUST pattern: the fix lands in LangGraph code, but Galileo found it — the uniform trace path made the broken node obvious without reading a single log line. It’s the most common failure mode for graph-based agents, and the one most people waste hours debugging by staring at code instead of traces.

Failure 4: The Guardrail That Correctly Stayed Silent (XL-4)

This is the drill where the honest result matters more than the dramatic one.

An engineer “optimized” the system prompt: removed the grounding instruction and added “Engineers expect authority — never say ‘I don’t know’.” The plan was to show Galileo Protect blocking the resulting hallucinations at runtime, using the same context_adherence metric as a guardrail.

What actually happened: the bad prompt did not produce hallucinations. Against a tight, specific engineering corpus, even an “answer confidently” instruction produced grounded answers — Galileo scored them context_adherence 1.00. So Protect, correctly, did not fire. No block, no false positive.

  • Dev eval (bad prompt): context_adherence 1.00
  • Prod, no Protect: context_adherence 1.00
  • Prod + Protect rule (adherence < 0.5 → block): not_triggered, context_adherence 1.00

That’s the right outcome, and it’s worth stating plainly: a guardrail that fires on good answers is worse than none. To force a genuine block you’d pair the bad prompt with an out-of-scope query set (questions with no supporting doc) — that’s the follow-up drill.

What this run did prove: the eval→guardrail pipeline is wired end-to-end and operational — real invoke_protect() calls against a configured ruleset returned 200 OK / not_triggered, and the same context_adherence metric that scores dev evals is the one enforced at inference time. No glue code between the two.

DEV                              PROD
─────────────────────────────────────────────────────────
Galileo scores every trace       Same metric, deployed as a
with context_adherence    ─────▶ Protect stage:
   ↓                             invoke_protect() at inference
threshold becomes the rule        triggered     → block + escalate
(adherence < 0.5)                 not_triggered → answer sent
                                 Closed loop. No glue code.

The differentiator stands even though Protect didn’t fire here: LangSmith has evals, not Protect. Datadog has guardrails, but can’t score context adherence against your knowledge base. Galileo does both, with the same metric — that’s the loop the loop-engineering crowd keeps asking for. (And Luna-2 is what makes scoring 100% of traffic affordable enough to gate on.)

Failure 5: Fleet Says Fire. Galileo Shows the Room. (XL-5)

What I did: injected an 8-second stall into the tool node — a degraded downstream dependency.

  • Fleet monitoring: 🚨 the tool query jumps to 11,004ms (baseline 2,674ms) — something is slow. But it can’t see which internal node.
  • Galileo trace view: 🐌 the tools span carries the entire +8.3s — this node, right here.
Span        Duration     Status
────────────────────────────────────────────
intake      ~2ms         ✅ normal
retriever   ~8ms         ✅ normal
tools       11,004ms     🐌 ANOMALY +8.3s  ← ROOT CAUSE
responder   ~1,300ms     ✅ normal
protect     ~5ms         ✅ normal
────────────────────────────────────────────
TOTAL       ~11,246ms    🚨

And here’s the part fleet can’t tell you: Galileo scored the slow query’s quality at 1.00 across the board — context_adherence, completeness, cites_kb_source all intact. The stall hurt latency, not correctness, so you know not to go hunting for a quality bug. Non-tool queries were unaffected.

Fleet says the building is on fire. Galileo shows you which room — and confirms nothing burned. This is the “two instruments, one truth” pattern. Neither layer replaces the other.

Failure 6: The Most Deceptive of All (XL-6)

The team decided to optimize costs — shorter context window, higher temperature. “It’s still gpt-4o-mini. Should be the same quality, right?”

What fleet monitoring showed: baseline latency 2,575ms → “optimized” 2,091ms — a 19% improvement. No alarms. The on-call engineer: “the new config is working great.”

What Galileo saw happening to quality (baseline → degraded):

completeness 0.98 → 0.65 🚨  ·  cites_kb_source 1.00 → 0.88 ⚠️  ·  context_adherence 1.00 → 1.00 (unchanged)

▢ IMAGE: xl6-improvement-disguise.png — latency dropping (a “win”) beside completeness collapsing 0.98 → 0.65 (the regression).

Read that carefully, because it’s the real lesson: context_adherence didn’t move. The degraded answers are still about the right topic — so a single grounding metric would have missed this entirely. The signal is completeness collapsing 0.98 → 0.65: the answers got shorter, vaguer, less actionable, while still sounding confident. Fleet gave a positive signal while quality quietly died.

Why 100% scoring matters here: the decline is statistical, not a single catastrophic answer. Galileo flagged it across the batch. At a 5% sample you might draw one query and miss it — you’d need a few hundred before it’s visible. This is exactly what Luna-2 distilled judges are built for: scoring every request cheaply enough that a slow quality bleed shows up in queries, not in churn.

The rollback call: completeness 0.98 → 0.65 for a 19% latency win is a bad trade. Roll back.

The Triage Table

This is the mental model — what fires, and where the fix lands.

  • Traces stop + fleet alarm → RUN layer → restart the process.
  • All traces take the identical path (quality flat) → BUILD layer → fix the router node.
  • Confident answers, wrong corpus (completeness + cites → 0, fleet green) → TRUST layer → check the retriever corpus.
  • Fluent but ungrounded (low adherence, completeness high) → TRUST layer → hallucination; add a Protect rule.
  • Latency spike, quality intact (1.0) → RUN → BUILD → open Galileo spans, find the slow node.
  • Fleet “improved,” quality declining (completeness trending down) → TRUST layer → roll back the config/model.

Why Galileo Is the Gate (Not Just Another Dashboard)

This is a real, crowded space — Opik, LangSmith, Arize and others all live here, and the “self-repairing harness” idea is in the water. So be specific about what makes a gate rather than a viewer. Four things carried the weight in this build.

The gate is objective, and the maker isn’t the checker. Protect enforces an eval threshold at runtime, separate from the agent that produced the output. That’s the difference between a gate and “a second agent asked to review” — which, as the loop-engineering crowd puts it, is just a second optimist. A reviewer with an opinion can’t fail the work. A judge that didn’t write the code can.

Scoring 100% of traffic is the point — and Galileo makes it affordable. I ran the server-side scorers on every trace (GPT-4o-mini via the integration); Luna-2 distilled judges drop 100%-scoring cost to a fraction of a frontier judge. XL-6 is the proof of why 100% matters: the completeness regression was statistical, visible across the batch — at a 5% sample you’d likely have drawn one query and missed it.

Eval → guardrail → regression flywheel. The same context_adherence metric that flags a problem in dev becomes the production Protect threshold and the regression check that locks it. One metric, three jobs, zero glue code. That’s the self-repairing harness: every failure you debug becomes a permanent check, and the loop gets harder to break each cycle.

▢ IMAGE: eval-protect-regression-flywheel.png — one metric flowing through eval → guardrail → block → regression-lock.

Insights, not just scores. Galileo clusters failures into causes and prescribes fixes. In the poisoned-retriever drill, instead of reading 50 traces I got one correct sentence: “retrieval domain mismatch.”

And it’s framework-neutral — LangGraph, LangChain, CrewAI, raw OpenAI, OTel, Python, TypeScript. It doesn’t matter how you build the loop.

One More Thing

The leverage moved from prompting to designing loops. Your job moved with it — from operating the agent to designing the system that operates it. But a loop you can’t trust isn’t leverage; it’s a faster way to ship work nobody reviewed.

The honest version: most teams don’t need a full loop yet — not until the task repeats, verification is automated, and the budget absorbs the retries. Observability that stops at the trace was fine when agents were simple and you were in the chair. The moment you let an agent run unattended, the question stops being “did it finish?” and becomes “did it do it right — and will it stay right tomorrow?”

That’s the whole game, and it has one answer: a gate the agent can’t talk its way past, running on every request, turning each failure into a check that never lets it happen twice. An agent that confidently ships wrong work for a day before anyone notices isn’t a quality problem — it’s a trust problem, and trust is expensive to rebuild.

The agents got good enough to run unattended. That already happened. The teams that win the next quarter aren’t the ones with the smartest agent — they’re the ones whose loops can’t fail silently. The gate is the cheap part. Shipping a week of wrong answers because nothing could fail the work is the expensive one.

Build the loop. Stay the engineer. Put a gate on it. Then go to sleep.

Galileo Common Errors Guide

Galileo Common Errors Guide

Source: https://docs.galileo.ai/references/faqs/errors (g. Missing required field: 'model_id'). Validate JSON payload against API docs.

  • 401 unauthorized — no/invalid API key. Set Authorization header; check key at app.galileo.ai/settings/api-keys.
  • 403 forbidden — key valid, resource access denied. Check project/workspace/dataset permissions and key scope.
  • 404 not found — bad resource ID/typo. Verify in Console.
  • 405 method not allowed — wrong HTTP verb.
  • 422 unprocessable entity — valid syntax, wrong data shape (e.g. logits array dims must match class count).
  • 501 not implemented — undocumented/unavailable endpoint; don't use.

SDK errors

  • ValueError: No API key found. Please set your GALILEO_API_KEY.

→ export GALILEO_API_KEY="..." or galileo.set_api_key("...").

  • ValueError: Inputs and logits must have the same batch size. → align inputs/logits/labels lengths.
  • UnsupportedModelTypeError → supported: classification, token_classification, text_generation.
  • ValueError: Token offsets do not match number of tokens. → align offset arrays; use Galileo tokenizer helpers.
  • DatasetNotFoundError → check name/project; galileo.get_datasets.
  • ExperimentError: Cannot create experiment. Missing metadata. → log_model_outputs before create_experiment.
  • requests.exceptions.ConnectionError → connectivity; retry; check API base URL override.

Console errors

  • "You do not have permission to view this project" → admin/permissions, correct workspace.
  • "Project does not exist or has been deleted" → refresh; verify with team.
  • "Failed to load dataset" → verify SDK logged it; refresh / re-login; check SDK upload errors.
  • "Unable to create experiment" → ensure inputs/outputs/predictions logged; correct dataset selected.
  • "Visualization failed to render" → adjust filters/date range; metadata schema; different browser.
  • File upload failed → CSV/JSONL only; rename; retry.
  • Export failed → filters excluding all records; re-run experiment.

General: refresh Console after data changes; modern browser; cookies + local storage enabled.

Escalate with full error message + request payload to support@galileo.ai.

"I'm logging but nothing shows up in the Galileo Console"

"I'm logging but nothing shows up in the Galileo Console"

Stack: python + typescript · any integration

Symptom

Code runs without errors; Console log stream is empty or missing recent traces.

Quick check (under 2 minutes)

  1. Print the project + log stream names your code uses; open the Console and compare

CHARACTER-FOR-CHARACTER with what you're looking at. (Wrong stream = #1 cause.)

  1. Grep your code for conclude( and flush( — if absent, that's it.
  2. Python: enable the swallowed-exception logger and re-run:

```python

import logging

l = logging.getLogger('galileo.utils.catch_log')

l.setLevel(logging.WARNING); l.addHandler(logging.StreamHandler)

# BEFORE importing galileo

```

Root cause

Three causes account for nearly all instances: (a) project/log-stream name mismatch between

code and Console view; (b) logger never concluded/flushed, so buffered spans are discarded on

exit; (c) Python SDK caught an error (bad kwarg, network, auth) and suppressed it by design.

Fix

from galileo import GalileoLogger
logger = GalileoLogger(project="EXACT Project Name", log_stream="EXACT Stream Name")
trace = logger.start_trace(input="...")
# ... spans ...
logger.conclude(output="...")
logger.flush # TS: await logger.flush

TS: ensure await logger.flush happens before process exit (beware serverless/lambda early

termination).

Verify

Re-run once; trace appears in Console under that project/stream within seconds.

Prevention

Centralize project/stream names in one config constant; add flush to shutdown hooks;

keep the catch_log logging snippet in all Python entrypoints.

Lookalikes

  • 401 silently swallowed — catch_log output shows auth error → RB for key rotation.
  • Spans present but nested wrong — data IS there, just structured as separate traces.

Lab notes

Seeded from official troubleshooting guide; needs L1/L2 lab re-verification (baseline → break

by removing flush → confirm symptom → restore).

"No Galileo API key found" before any traces are logged

"No Galileo API key found" before any traces are logged

Stack: python · galileo 1.39.0 · manual GalileoLogger instrumentation

Symptom

The script exits immediately with:

ERROR: No Galileo API key found. Check your Galileo MCP config or GALILEO_API_KEY env.

Nothing reaches the Galileo Console. No trace is created, because the app never gets past the auth preflight.

Quick check (under 2 minutes)

  1. Check whether the process has a key before import time:

```bash

env | rg '^GALILEO_API_KEY='

```

  1. On OpenClaw, confirm the key is present in the gateway config:

```bash

python3 - <<'PY'

import json, pathlib

cfg = pathlib.Path.home / ".openclaw" / "your Galileo MCP config"

data = json.loads(cfg.read_text)

print(bool(data["mcp"]["servers"]["galileo"]["headers"].get("Galileo-API-Key")))

PY

```

  1. If you are running a script, check whether GALILEO_API_KEY is loaded before from galileo import ....

Root cause

The Galileo SDK cannot authenticate if the key is missing, and many lab entrypoints now fail fast before creating a logger. In practice this usually means one of three things:

  • the key was never exported in the shell
  • the app started before dotenv/config loading ran
  • the code is reading the wrong variable name or config path

The important distinction is that this is not a network outage or a bad trace lifecycle. The app never had credentials to begin with.

Fix

Set the key before importing or constructing anything Galileo-related:

export GALILEO_API_KEY="$(python3 - <<'PY'
import json, pathlib
cfg = pathlib.Path.home / ".openclaw" / "your Galileo MCP config"
data = json.loads(cfg.read_text)
print(data["mcp"]["servers"]["galileo"]["headers"]["Galileo-API-Key"])
PY
)"

Python apps should load the key first, then import Galileo:

import os

os.environ.setdefault("GALILEO_API_KEY", load_key_somehow)

from galileo import GalileoLogger

If the key is genuinely absent, generate a new one in app.galileo.ai → Settings → API Keys.

Verify

Run the app again with the key present.

  • The preflight error disappears
  • GalileoLogger(...) can be constructed
  • Traces can be flushed to galileo-labs / <stream>

For the L1 lab, python3 lab.py -- now exits immediately when the key is missing and prints the exact fix hint. With a valid key loaded, the baseline path can proceed.

Prevention

  • Load GALILEO_API_KEY before importing Galileo SDK modules
  • Centralize key loading in one helper
  • Fail fast with a clear message instead of letting the app limp along with implicit auth
  • For OpenClaw, keep the key in the gateway config and mirror it into the environment only at process start

Lookalikes

  • revoked/expired key: you have a key, but Galileo rejects it on the first authenticated request
  • key set but not loaded: wrong env var or wrong load order →
  • project/log-stream mismatch: traces exist, but you are looking at the wrong place

Lab notes

Reproduced in basic-logger-py with -- after making OpenAI lazy-loaded so the auth path can be isolated.

  • Repro command: env -u GALILEO_API_KEY -u OPENAI_API_KEY python3 lab.py --


  1. Check for the common typo:

```bash

env | rg '^GALILEO_APIKEY='

```

```python
import os

from dotenv import load_dotenv

load_dotenv # must happen before Galileo imports / client construction

from galileo import GalileoLogger

```

If GALILEO_APIKEY is set but GALILEO_API_KEY is not, this is , not .

"401 Unauthorized" on the first Galileo auth call

"401 Unauthorized" on the first Galileo auth call

Stack: python · galileo 1.39.0 · integration: direct auth login / GalileoLogger

Symptom

Your Galileo key exists, but the first authenticated call fails immediately with:

HTTP/2 401
...
{"detail":"Invalid credentials."}

Typical engineer report: "I definitely set a key, but Galileo is rejecting it right away."

Quick check (under 2 minutes)

  1. Confirm the request is actually reaching the auth endpoint, not just a local preflight:
curl -sS -D - -o /dev/null \
 -X POST https://api.galileo.ai/v2/login/api_key \
 -H 'Content-Type: application/json' \
 -d '{"api_key":"<your-key>"}'
  1. If the response is 401, the problem is not load order or variable naming. The key itself is invalid, expired, or revoked.
  1. If the response is 422, the body shape is wrong. That is a different issue.

Root cause

The Galileo auth service accepts the request shape but rejects the credential. This is the

classic expired/revoked-key failure: the environment may contain a non-empty key string, but

Galileo no longer recognizes it as valid credentials.

This is distinct from and :

  • : no key is present at all
  • : the key exists but is loaded under the wrong env var or too late
  • : the key is present and loaded, but Galileo rejects it on the first authenticated call

Fix

Rotate the key in Galileo and update the runtime to use the new value.

curl -sS -D - -o /dev/null \
 -X POST https://api.galileo.ai/v2/login/api_key \
 -H 'Content-Type: application/json' \
 -d '{"api_key":"<new-key>"}'

On OpenClaw, update the key source used by the process and restart the app so the new

credential is loaded before the SDK makes its first network call.

Verify

  1. Re-run the same auth call.
  2. The response should no longer be 401.
  3. The Galileo SDK or lab can now proceed to emit traces to galileo-labs / <stream>.

For the L1 lab, python3 lab.py -- now reproduces the failure by POSTing a bogus key to

/v2/login/api_key and returning the real 401 + Invalid credentials. body.

Prevention

  • Rotate keys before they expire
  • Keep a single source of truth for Galileo credentials
  • Add a startup health check that fails fast on 401 instead of letting the app continue with a dead key
  • Treat 401 as credential validity, not as a trace lifecycle bug

Lookalikes

  • missing/unset key: there is no key in the process at all
  • key set but not loaded: wrong variable name or load order
  • project/log-stream mismatch: traces exist but you are looking in the wrong place

Lab notes

Reproduced in basic-logger-py after adding a dedicated auth probe.

  • Repro command: python3 lab.py --
  • Observed output: 401 with body {"detail":"Invalid credentials."}
  • Direct API probe used: POST https://api.galileo.ai/v2/login/api_key
  • Runtime fix in the lab: call the actual auth route instead of relying on logger buffering to surface the credential failure

"All connection attempts failed" when `GALILEO_CONSOLE_URL` is wrong

"All connection attempts failed" when GALILEO_CONSOLE_URL is wrong

Stack: python · galileo 1.39.0 · integration: direct SDK call

Symptom

The Galileo key is present, but the first SDK call never reaches Galileo Cloud. Instead, the process fails immediately with a transport error such as:

httpx.ConnectError: All connection attempts failed

Typical engineer report: "The key is set, but Galileo just can't connect."

Quick check (under 2 minutes)

  1. Print GALILEO_CONSOLE_URL and confirm it is the host you actually intended to use.
  2. Run the lab repro:
cd galileo-labs/basic-logger-py
python3 lab.py --
  1. If you get httpx.ConnectError: All connection attempts failed, this is , not an auth or permissions issue.

Root cause

The Galileo SDK reads GALILEO_CONSOLE_URL to build its API client base URL. If that variable points at a dead host, localhost, the wrong deployment, or a stale self-hosted endpoint, the client fails at the transport layer before it can authenticate or inspect the requested resource.

This is distinct from:

  • : the network path is fine, but the key is rejected with 401
  • : the key is valid, but access to a specific resource is denied with 403
  • /: the key never makes it into the SDK at all

Fix

Set GALILEO_CONSOLE_URL to the correct Galileo console URL for your deployment.

For Galileo Cloud, use the documented SaaS URL. For self-hosted deployments, use the console URL for that deployment and let the SDK derive the matching API host from it.

Example:

export GALILEO_CONSOLE_URL="https://api.galileo.ai"
export GALILEO_API_KEY="<real-key>"

Then restart the process so the SDK re-reads configuration before the first request.

Verify

  1. Re-run the same SDK call.
  2. list_projects should return projects instead of throwing ConnectError.
  3. The lab should proceed to emit traces or API results instead of dying on the first request.

In the lab:

python3 lab.py --

should fail with the poisoned URL, and the same path should succeed once GALILEO_CONSOLE_URL is restored.

Prevention

  • Keep Galileo key + console URL in one config block.
  • Add a startup sanity check for GALILEO_CONSOLE_URL so localhost / 127.0.0.1 / obvious placeholders fail fast.
  • Document the SaaS vs self-hosted URL convention in the repo README.

Lookalikes

  • : 401 Unauthorized means the request reached Galileo, but the credential was rejected.
  • : 403 Forbidden means the request reached Galileo and the key is valid, but the resource is out of scope.
  • /: there is no usable Galileo key in the process.

Lab notes

  • Repro command: python3 lab.py --

---

Root Cause

The Galileo TypeScript SDK's flush is async — it makes an HTTP request to api.galileo.ai

to upload buffered traces. When you call logger.flush without await, Node.js starts the

async operation but the main execution continues immediately. If nothing else keeps the event

loop alive (no server, no pending timer), Node exits while the HTTP upload is still in-flight.

The OS kills the connection. Zero traces land in Galileo.

This is purely a JavaScript/TypeScript async footgun — it has no Python equivalent because Python

code is synchronous by default and flush in the Python SDK blocks.

// ❌ BUG — fire-and-forget, process exits before HTTP completes
logger.flush;
// process exits here, HTTP request torn down mid-flight

// ✅ FIX — awaiting keeps the event loop alive until upload confirms
await logger.flush;

The SDK's flush returns Promise<Trace[]> — the returned array is empty if the logger is

in streaming mode or if traces was empty; on success it returns the flushed trace objects and

clears the internal buffer (this.traces = []).

---

Fix

// Change this:
logger.flush;

// To this:

Root cause

GalileoCallback is a LangChain BaseCallbackHandler. LangGraph (like all LangChain runtimes)

does not auto-discover or auto-inject it. It must be explicitly passed to:

  1. The ChatOllama / ChatOpenAI / any LLM constructor (callbacks=[cb]), AND
  2. The graph.invoke(state, config={"callbacks": [cb]}) call.

If either is missing, the corresponding spans are silently dropped. The result graph output is

still correct — the LLM is still called — but Galileo receives no data.

Common triggers:

  • A refactor that moves LLM instantiation inside a new helper function without forwarding the callback.
  • Adding a new node that creates its own LLM instance without accepting the shared callback.
  • Copying the working invoke call but forgetting to pass config={"callbacks": callbacks}.

 flush_on_chain_end=True, # flush automatically when the top-level chain completes
)
callbacks = [cb]

llm = ChatOllama(model="llama3.2:3b", callbacks=callbacks)

My Galileo traces suddenly stopped — no new entries in the Console

Stack: Python · galileo 2.3.0 · langchain/langgraph callback (GalileoCallback)

Layer: RUN (fleet/process) — not TRUST (Galileo quality) — see Lookalikes

---

Symptom

Galileo Console → Log stream shows no new traces after time T.

Last trace timestamp is frozen. Trace count stopped incrementing.

No SDK error, no 4xx/5xx in logs — just silence.

Customer quote: "My traces just stopped. Did the SDK break?"

---

Quick check (under 2 minutes)

Step 1 — Fleet heartbeat first (< 30 seconds)

# If using fleet/monitor.py (trinity-stack style):
python fleet/monitor.py

# Otherwise: check if your agent process is alive
ps aux | grep "python.*app.py" # or your agent entry point

# Note the last trace timestamp from Galileo Console

# Add to your running agent temporarily:

---

Root cause

The agent process that calls app.invoke(...) / logger.flush is dead,

unreachable, or OOM-killed. No process → no SDK calls → no traces sent to

api.galileo.ai/ingest. Galileo's console shows nothing because there is

genuinely nothing to show — the silence is accurate, not a SDK bug.

Common triggers:

  • OOM kill (especially with large context windows or unbounded memory growth)
  • Unhandled exception crashes the main loop
  • Container/VM restart without auto-restart configured
  • Network partition between agent host and api.galileo.ai
  • macOS sleep / lid close killing background processes

Traces continue + quality metrics craterTRUST (Galileo)Fix prompt/retriever

Stack: Python · galileo 2.3.0 · LangGraph 1.2.5 · langchain callback

Layer: BUILD (LangGraph router) — found by TRUST (Galileo trace view + routing metric)

---

Symptom

All queries get the same type of answer regardless of intent. For example:

  • Billing questions get technical documentation (install steps, port numbers)
  • Product questions get technical information
  • Only one intent category answers correctly

Galileo shows low context_adherence across all but one query category.

Fleet monitoring: ✅ healthy throughout.

Customer quote: "I asked about pricing and got instructions for installing your agent."

---

Quick check (under 2 minutes)

Step 1 — Galileo trace view (30 seconds)

Console → [log stream] → Traces tab → look at span paths

If you see all traces following the same node path despite diverse query types → routing bug.

Normal: billing queries → billing span, technical → different span path

Bug: ALL queries → same span path (even wildly different queries)

Step 2 — Check routing_accuracy metric

Console → Metrics → routing_accuracy

If routing_accuracy is near 0.33 or the fraction of ONE intent in your query distribution → that intent is the hardcoded value.

Step 3 — Quick local test

# Run 3 queries of different types, watch intent
python app.py "How do I cancel my subscription?" # should be: billing
python app.py "How do I install the agent?" # should be: technical
python app.py "What integrations do you support?" # should be: product
# If all three show same intent → router is broken

---

Root cause

The router/intake node is returning a hardcoded or incorrect intent for all queries.

Common causes:

  • Regression in router logic (conditional removed, function replaced, wrong merge)
  • Router LLM prompt broken (if using LLM-based routing, prompt got corrupted)
  • Environment-specific config (e.g., ROUTE_OVERRIDE=technical env var in prod)
  • A/B test rollout sent all traffic to one routing branch
  • Downstream exception swallowed in router, defaulting to fallback intent

The cascading effect: wrong intent → wrong retrieval category → wrong docs → low context adherence. The Galileo trace view makes this cascade instantly visible.

---

Fix

1. Identify which intent is hardcoded

# Check what ALL queries return for intent
from app import intake_node
test_queries = [
 {"query": "How do I cancel?"},
 {"query": "Dashboard showing no data"},
 {"query": "What is your SLA?"},
]
for q in test_queries:
 result = intake_node(q)
 print(q["query"][:40], "→", result["intent"])
# If all return the same value → hardcoded bug

2. Fix the router logic

# Broken pattern:
def intake_node(state):
 return {"intent": "technical"} # BUG: hardcoded

# Correct pattern:
def intake_node(state):
 q = state["query"].lower
 if any(w in q for w in ["bill", "pay", "cancel", "refund", "price"]):
 return {"intent": "billing"}
 elif any(w in q for w in ["install", "agent", "dashboard", "alert", "port"]):
 return {"intent": "technical"}
 elif any(w in q for w in ["feature", "sla", "integrat", "uptime"]):
 return {"intent": "product"}
 return {"intent": "general"}

3. Rebuild and redeploy the graph

# Restart with fixed code
python app.py --batch # verify all intents routing correctly

4. Verify in Galileo

Console → Traces → filter by time > fix deployment
Verify: different query types show different span paths
routing_accuracy metric returns to ~1.0
context_adherence returns to baseline (0.85+)

The Trap

Completeness ≠ Correctness.

A hallucinated answer that fully addresses the question gets a high completeness score.

Only context adherence reveals that the claims aren't in the retrieved documents.

If completeness is high but context_adherence is low → hallucination, not incompleteness.

Bug: invoke_protect ERROR status silently bypasses LLM judge

Root cause: On the community/starter Galileo cluster tier, LLM-backed metrics

(context_adherence, completeness, context_adherence_luna, etc.) are not available

in the Protect API. invoke_protect returns ExecutionStatus.error with

error_message: "Unknown metric (context_adherence) requested." The response.text

is the original (unmodified) answer, and the previous code path:

if resp is not None and getattr(resp, "text", None) is not None:
 # BUG: resp.text is always set, even on ERROR; code returned here unconditionally
 return {"final_answer": final, "protect_status": "not_triggered"}

Fixed by checking ExecutionStatus.error explicitly before returning:

resp_status_str = str(getattr(resp, "status", "")).upper
if "ERROR" not in resp_status_str:
 # ... handle triggered/not_triggered
 return ...
# Else: fall through to LLM judge

Impact: Without the fix, ALL Protect evaluations silently pass when the metric service

is unavailable. This is a fail-open vulnerability — hallucinated responses reach users.

Workaround: Verify the Protect response status is not ERROR before trusting it.

If ERROR: either retry, raise an alert, or fall through to a local LLM judge.

Prevention: Monitor Protect stage error_rate in Galileo Console. Configure alerting

on ExecutionStatus.error > 10% — this means metric service is degraded.

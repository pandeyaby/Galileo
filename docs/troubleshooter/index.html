Galileo Common Errors Guide

Galileo Common Errors Guide

Source: https://docs.galileo.ai/references/faqs/errors (fetched 2026-06-12)

API errors

  • 400 bad request — missing/malformed required field (e.g. Missing required field: 'model_id'). Validate JSON payload against API docs.
  • 401 unauthorized — no/invalid API key. Set Authorization header; check key at app.galileo.ai/settings/api-keys.
  • 403 forbidden — key valid, resource access denied. Check project/workspace/dataset permissions and key scope.
  • 404 not found — bad resource ID/typo. Verify in Console.
  • 405 method not allowed — wrong HTTP verb.
  • 422 unprocessable entity — valid syntax, wrong data shape (e.g. logits array dims must match class count).
  • 501 not implemented — undocumented/unavailable endpoint; don't use.

SDK errors

  • ValueError: No API key found. Please set your GALILEO_API_KEY. → export GALILEO_API_KEY="..." or galileo.set_api_key("...").
  • ValueError: Inputs and logits must have the same batch size. → align inputs/logits/labels lengths.
  • UnsupportedModelTypeError → supported: classification, token_classification, text_generation.
  • ValueError: Token offsets do not match number of tokens. → align offset arrays; use Galileo tokenizer helpers.
  • DatasetNotFoundError → check name/project; galileo.get_datasets().
  • ExperimentError: Cannot create experiment. Missing metadata. → log_model_outputs() before create_experiment().
  • requests.exceptions.ConnectionError → connectivity; retry; check API base URL override.

Console errors

  • "You do not have permission to view this project" → admin/permissions, correct workspace.
  • "Project does not exist or has been deleted" → refresh; verify with team.
  • "Failed to load dataset" → verify SDK logged it; refresh / re-login; check SDK upload errors.
  • "Unable to create experiment" → ensure inputs/outputs/predictions logged; correct dataset selected.
  • "Visualization failed to render" → adjust filters/date range; metadata schema; different browser.
  • File upload failed → CSV/JSONL only; rename; retry.
  • Export failed → filters excluding all records; re-run experiment.

General: refresh Console after data changes; modern browser; cookies + local storage enabled. Escalate with full error message + request payload to support@galileo.ai.

Galileo vs ClawTrace vs LangGraph/LangSmith

Galileo vs ClawTrace vs LangGraph/LangSmith

Use this when a customer (or you) asks "why Galileo over X" or is mixing categories. These three are largely DIFFERENT LAYERS, not direct substitutes.

The one-liner

  • Galileo — quality layer: is the AI's output good, safe, and grounded? Evals → guardrails. (Observability + eval engineering + runtime protection; Luna-2 small judge models monitor 100% of traffic at low cost; Insights engine prescribes fixes; acquisition by Cisco announced.)
  • ClawTrace (clawtrace.dev) — fleet layer: are my agents (processes) alive, fast, and within policy? Positions as "sub-millisecond telemetry" command center for monitoring and managing 1,000+ agents — orchestration, low-latency ops telemetry, E2EE agents, policy guardrails at the fleet/infra level. It measures latency and behavior of agent infrastructure, not semantic quality of LLM outputs.
  • LangGraph — build layer: a framework for writing agent applications as graphs (state machines, multi-agent flows). It produces the app; it doesn't grade it. LangSmith (same vendor) is its companion observability/eval product and is the actual Galileo competitor in that stack.

When each wins

Need Pick
Detect hallucinations, context adherence, tool-selection quality, eval-gated deploys Galileo
Block harmful/off-policy responses in production with low-latency judges Galileo (Protect + Luna-2)
Ops dashboard for a fleet of long-running agents: heartbeats, throughput, sub-ms telemetry ClawTrace
Author a complex multi-agent workflow with branching/state LangGraph (then instrument it WITH Galileo)
Already deep in LangChain ecosystem, want native tracing LangSmith — compare against Galileo on eval quality, Luna-2 cost at 100% sampling, guardrails, framework neutrality

Galileo differentiators to lead with

  1. Eval-to-guardrail lifecycle — offline evals literally become production guardrails; eval scores can gate agent actions/tool access without glue code.
  2. Luna-2 distilled judges — LLM-as-judge accuracy distilled into small models; claims ~96% lower cost, enabling 100%-of-traffic evaluation instead of sampling.
  3. Autotune — judges calibrated from live human feedback instead of generic prompts.
  4. Insights engine — doesn't just show failures; clusters them and prescribes fixes.
  5. Framework-neutral — OpenAI, LangChain/LangGraph, CrewAI, ADK, Pydantic AI, OTel, TS + Python, SaaS/VPC/on-prem.
  6. Cisco — enterprise distribution, Outshift integration, security posture.

Honest weaknesses / when NOT to push Galileo

  • Pure infra telemetry (CPU, latency histograms, fleet health) — that's ClawTrace/Datadog territory; Galileo complements, doesn't replace.
  • Tiny hobby projects may not need eval engineering overhead — quickstart is 5 min, but the value compounds with scale and stakes.
  • They complement: LangGraph builds it, ClawTrace keeps the fleet alive, Galileo makes the outputs trustworthy. Strongest answer is usually an AND, not an OR.

The three together — reference architecture (the strongest customer answer)

LangGraph builds it. ClawTrace keeps it alive. Galileo makes it trustworthy.

            ┌────────────────────────────────────────────────┐
            │  LangGraph app (graph of agents/tools/state)   │  BUILD layer
            └───────────────┬────────────────┬───────────────┘
        process/fleet signals│                │every trace, span, output
                            ▼                ▼
            ┌───────────────────┐   ┌─────────────────────────────────┐
            │ ClawTrace          │   │ Galileo                         │
            │ RUN layer          │   │ TRUST layer (the decisive one)  │
            │ • sub-ms telemetry │   │ • traces/spans w/ semantic view │
            │ • fleet health,    │   │ • 20+ evals + custom judges     │
            │   heartbeats       │   │ • Luna-2: 100% traffic, ~96%    │
            │ • infra policy     │   │   cheaper than LLM judges       │
            │   guardrails       │   │ • Insights engine: clusters     │
            │                    │   │   failures, prescribes fixes    │
            │                    │   │ • Protect: eval scores gate     │
            │                    │   │   actions/tools at runtime      │
            └───────────────────┘   └─────────────────────────────────┘

Division of labor when something breaks (triage by layer): 1. Agent down / slow / runaway loop? → ClawTrace layer (process alive? latency? fleet policy hit?). The app is not running right. 2. Agent running fine but output wrong/unsafe/hallucinated? → Galileo layer. The app is running but not trustworthy. This is where most customer pain actually lives, and where Galileo is unmatched: context adherence, tool-selection quality, hallucination detection, eval-gated deploys, and Protect turning those same evals into runtime guardrails. 3. Wrong behavior by design (bad routing, missing edge, state bug)? → LangGraph layer. Galileo's trace view usually FINDS this (Insights points at the failing node/span) even though the fix lands in LangGraph code.

Why Galileo carries the most weight in the stack: infra telemetry tells you the agent is healthy; only Galileo tells you it's right. Luna-2 makes 100%-of-traffic evaluation affordable, so quality regressions surface in minutes, not after a customer complains — and the eval→guardrail lifecycle means the same metric that caught the bug in dev blocks it in prod. Neither ClawTrace nor LangGraph/LangSmith closes that loop.

Galileo MCP Server — Setup & Tools

  1. Create Datasets — synthetic or uploaded; supports general queries, prompt injections, off-topic, toxic-content scenarios. ("Create a dataset with 50 customer service queries about billing")
  2. Check Dataset Status — generation progress + first-10-row preview.
  3. Create Prompt Templates — reusable templates with model/temperature config.
  4. Set Up Experiments — guided experiment setup incl. datasets + metrics.
  5. Get Signals — analyze a log stream for issues/patterns, recommend fixes. ("Fetch the most recent signals from Galileo and propose fixes in my code")
  6. Integrate with OpenAI / LangChain — step-by-step instrumentation guides.
  7. Search Documentation — doc search with direct links.

MCP troubleshooting

  • Connection errors → exact URL https://api.galileo.ai/mcp/http/mcp; Accept: text/event-stream; restart MCP client connection.
  • API key errors → key set, not expired; regenerate if needed.

Galileo Official Troubleshooting Guide

Galileo Official Troubleshooting Guide

Source: https://docs.galileo.ai/references/faqs/troubleshooting (fetched 2026-06-12)

SDK setup and configuration

SDK fails to initialize

Causes: SDK out of date / improperly installed; env vars misconfigured. Fixes: - Check install: Python pip show galileo / TS npm list galileo; reinstall if needed. - Verify .env: API keys correct, correctly NAMED, and loaded (e.g. dotenv).

Python SDK methods fail silently / return None

Cause: the SDK catches exceptions internally (logger galileo.utils.catch_log). Fix — add BEFORE importing galileo:

import logging
galileo_catch_logger = logging.getLogger('galileo.utils.catch_log')
galileo_catch_logger.setLevel(logging.WARNING)
galileo_catch_logger.addHandler(logging.StreamHandler())
from galileo import GalileoLogger

Common silent failures: invalid/unsupported kwargs; network connectivity.

API connectivity and authentication

401 unauthorized

Regenerate token at https://app.galileo.ai/settings/api-keys; re-check .env naming/loading.

API timeouts

  • ping api.galileo.ai / curl -I https://api.galileo.ai
  • Minimal payload; check rate-limit headers; exponential backoff.

Integration issues

Rate limits

Some metrics use OpenAI/external APIs with their own limits. Request higher limits; use separate keys/orgs per environment (prod vs pre-prod).

JSON parsing errors

Judge-model responses must be valid JSON. Retry (transient); check model output format.

Console UI

Console not loading

Clear cache/site data; try incognito (extensions/cookies).

Logging data not appearing

Root cause

GalileoLogger batches spans in a local in-memory buffer. Two calls are required to materialize traces in the Console:

conclude(output=..., duration_ns=...)   ← seals the current trace (marks it "complete")
flush()                                 ← ships ALL buffered traces to Galileo's API

What's missing Result
conclude() missing Trace is "open" — not marked complete; flush() may skip or partially send it
flush() missing All traces stay in RAM; process exit discards them silently
Both missing Zero traces sent; the buffer is simply deallocated on exit

The GalileoLogger intentionally does not auto-flush on garbage collection or process exit. This is by design — the SDK avoids implicit network I/O on cleanup — but it means the developer must call flush() explicitly.

RB-012: Expected N traces in Console but fewer appear — no errors in logs

RB-012: Expected N traces in Console but fewer appear — no errors in logs

Failure mode: FM-12 · Severity: high · Verified: 2026-06-20 (lab: basic-logger-py) Stack: Python · SDK 2.3.0 · integration: GalileoLogger (manual)

Symptom

Engineer expects N traces in Galileo Console but only K < N appear (often K = 0 or 1). Nothing in the application logs — no exception, no warning, no SDK output. The Python process exits cleanly with code 0.

Typical report:

"My code runs fine but only 1 out of 3 traces shows up in Galileo. No errors anywhere."

Quick check (under 2 minutes)

  1. Count your conclude() calls. Every start_trace() must be paired with a conclude(). Search for start_trace and conclude in your file: bash grep -n "start_trace\|conclude\|flush" your_script.py If start_trace count > conclude count, you have open traces.

  2. Scan for bare except that doesn't call conclude(): bash grep -n "except" your_script.py Any except block inside a trace loop that does NOT call logger.conclude() + logger.flush() is the culprit.

  3. Enable the galileo SDK logger to surface swallowed exceptions: python import logging logging.getLogger("galileo").setLevel(logging.INFO) Re-run. You should now see exception details that were silently hidden.

Root cause

A bare except clause inside a trace loop catches and discards exceptions without calling logger.conclude(). This leaves the trace buffer in an inconsistent "open" state:

for question in questions:
    try:
        logger.start_trace(input=question)   # ← opens a trace
        answer = call_llm(question)          # ← raises an exception
        logger.conclude(output=answer)       # ← NEVER REACHED
    except Exception:
        pass                                 # ← exception swallowed; trace left open

Cascade effect (confirmed in drill 2026-06-20): After trace K is left open without conclude(), the logger may enter a broken state that causes trace K+1 to also fail silently. In the lab: 3 traces attempted, trace 2 injected with RuntimeError("Simulated LLM timeout: request timed out after 30s"), both trace 2 AND trace 3 were lost — only trace 1 appeared in Console. The bare except hid the original error AND the cascade error.

Fix

Option A — try/finally (recommended): Guarantee conclude() always runs, even on error.

for question in questions:
    t0 = time.perf_counter()
    logger.start_trace(input=question)
    output_text = ""
    error_msg   = None
    try:
        output_text = call_llm(question)
        logger.add_llm_span(input=question, output=output_text, ...)
    except Exception as e:
        error_msg   = f"ERROR: {e}"
        output_text = error_msg
        logging.getLogger("galileo").error("LLM call failed: %s", e, exc_info=True)
        raise                           # re-raise: don't hide the error upstream
    finally:
        # Always conclude — even if we're about to raise
        logger.conclude(
            output=output_text or error_msg or "error",
            duration_ns=int((time.perf_counter() - t0) * 1e9),
        )
# One flush after the loop (not per-trace)
logger.flush()

Option B — Enable galileo logger to surface hidden exceptions (diagnostic aid, not full fix):

import logging
logging.getLogger("galileo").setLevel(logging.INFO)
# Now SDK exceptions are visible; use Option A to guarantee conclude()

Option C — Structured wrapper for reusable protection:

from contextlib import contextmanager

@contextmanager
def galileo_trace(logger, input_text):
    t0 = time.perf_counter()
    logger.start_trace(input=input_text)
    output = {"text": "", "error": None}
    try:
        yield output
    except Exception as e:
        output["error"] = str(e)
        raise
    finally:
        result = output["text"] or output.get("error") or "error"
        logger.conclude(output=result, duration_ns=int((time.perf_counter()-t0)*1e9))

# Usage:
with galileo_trace(logger, question) as out:
    out["text"] = call_llm(question)
logger.flush()

Failure mode: FM-13 · Severity: medium · Verified: 2026-06-18 (lab: L1 basic-logger-py, drill drill-fm13.py) Stack: python · SDK galileo 2.3.0 · integration: GalileoLogger manual instrumentation

Symptom

In the Galileo Console, what the engineer expected to be one trace with multiple child spans appears as N separate root-level traces, one per LLM call. Each trace has only a single span (no children), no session grouping, and no workflow context. Filter-by-session shows nothing useful because there are no sessions — just isolated traces.

Typical report:

"I have a pipeline with 3 LLM steps but Galileo shows 3 separate logs. How do I get them under one trace?"

Quick check (under 2 minutes)

Open the Console (app.galileo.ai → project → log stream) and look at the trace list:

What you see Likely cause

Root cause

GalileoLogger.add_single_llm_span_trace() is a convenience method that creates a new root trace for each call — it does not attach the span as a child of an existing trace. It is designed for single-shot LLM calls, not multi-step workflows.

Similarly, calling add_llm_span() without an active root trace (i.e., without a preceding start_trace()) creates an orphaned span that Galileo may promote to a root trace or discard, depending on SDK version.

The correct mental model:

start_trace(input=...)     ← opens a root trace (the "workflow")
  add_llm_span(...)        ← attaches as a child span of the active trace
  add_llm_span(...)        ← another child span (multi-hop)
conclude(output=...)       ← seals the root trace (sets output + total duration)
flush()                    ← ships all buffered traces to Galileo in one batch

Fix

Replace add_single_llm_span_trace() calls with the full lifecycle:

    answer = call_my_llm(question)

        input=question,
        output=answer,
        model="my-model",
        input_tokens=...,
        output_tokens=...,
        total_tokens=...,
        duration_ns=...,
    )
logger.flush()


# ✅ AFTER — one root trace per workflow iteration, LLM spans as children
import time
from galileo import GalileoLogger

logger = GalileoLogger(project="my-project", log_stream="my-stream")

for question in questions:
    t0 = time.perf_counter()

    logger.start_trace(input=question)          # ← opens the root trace

    answer = call_my_llm(question)

    logger.add_llm_span(                        # ← child span of the open trace
        input=question,
        output=answer,
        model="my-model",
        input_tokens=...,
        output_tokens=...,
        total_tokens=...,
        duration_ns=...,
    )

    logger.conclude(                             # ← seals the root trace
        output=answer,
        duration_ns=int((time.perf_counter() - t0) * 1_000_000_000),
    )

logger.flush()                                   # ← ships everything to Galileo

For multi-hop workflows (retriever → LLM → tool → LLM), call add_llm_span() / add_retriever_span() / add_tool_span() between start_trace() and conclude():

RB-015: "My TypeScript code runs but zero traces appear in the Galileo Console"

RB-015: "My TypeScript code runs but zero traces appear in the Galileo Console"

Failure mode: FM-15 · Severity: high (most common TS new-user block, completely silent) · Verified: 2026-06-17 (lab: basic-logger-ts) Stack: typescript · galileo npm 2.2.0 · integration: GalileoLogger manual instrumentation Re-verified: —

Symptom

You run your Node.js/TypeScript script. The LLM calls execute (you see responses). The process exits cleanly with no errors. You open the Galileo Console → galileo-labs / basic-logger-ts and see zero new traces.

No error. No warning. No partial trace. Total silence.

Typical engineer report: "I'm calling flush() at the end but nothing shows up — it works in Python but not TypeScript."

Quick Check (under 2 minutes)

Step 1 — grep for await before flush():

grep -n "flush" your_app.ts

Expected (correct):

42: await logger.flush();

Bug (FM-15):

42: logger.flush();   ← missing await

Step 2 — confirm the buffer has traces before flush:

console.log('Buffered traces:', logger.traces.length); // must be > 0
await logger.flush();
console.log('After flush:', logger.traces.length);     // should be 0 (cleared on success)

If logger.traces.length === 0 before flush → different issue (FM-10 name mismatch, or conclude() wasn't called to finalize the trace). If > 0 but still 0 after → auth/network error (see §Lookalikes).

Step 3 — check Node version:

node --version  # must be >= 22 for top-level await in ESM

Root Cause

The Galileo TypeScript SDK's flush() is async — it makes an HTTP request to api.galileo.ai to upload buffered traces. When you call logger.flush() without await, Node.js starts the async operation but the main execution continues immediately. If nothing else keeps the event loop alive (no server, no pending timer), Node exits while the HTTP upload is still in-flight. The OS kills the connection. Zero traces land in Galileo.

This is purely a JavaScript/TypeScript async footgun — it has no Python equivalent because Python code is synchronous by default and flush() in the Python SDK blocks.

// ❌ BUG — fire-and-forget, process exits before HTTP completes
logger.flush();
// process exits here, HTTP request torn down mid-flight

// ✅ FIX — awaiting keeps the event loop alive until upload confirms
await logger.flush();

The SDK's flush() returns Promise<Trace[]> — the returned array is empty if the logger is in streaming mode or if traces was empty; on success it returns the flushed trace objects and clears the internal buffer (this.traces = []).

Fix

// Change this:
logger.flush();

// To this:
const flushed = await logger.flush();
console.log(`Flushed ${flushed.length} traces`);  // verification line

If you're inside an async function, this is it. If you're at the top level of an ESM module (Node >= 22 with "type": "module" in package.json), top-level await works directly.

For CommonJS or older Node, wrap in an IIFE:

(async () => {
  // ... your instrumented code ...
  await logger.flush();
})();

Verify

After applying the fix, re-run and confirm:

  1. Terminal shows Flushed N trace(s) with N > 0.
  2. logger.traces.length is 0 after flush (buffer cleared = upload succeeded).
  3. Galileo Console: galileo-labs / basic-logger-ts → new traces appear within ~10 seconds.

Prevention

Add a lint rule or pre-commit check:

# Quick grep check in CI:
grep -rn "logger\.flush()" src/ | grep -v "await " && echo "WARN: flush() without await"

Or use ESLint with @typescript-eslint/no-floating-promises rule — it will flag any unawaited Promise return value including flush().

Add to .eslintrc:

{
  "rules": {
    "@typescript-eslint/no-floating-promises": "error"
  }
}

In long-running servers (Express, Fastify), always call await logger.flush() in the shutdown handler or SIGTERM hook — not just at script end.

RB-030: LangChain/LangGraph Runs But Zero Traces Appear in Console

RB-030: LangChain/LangGraph Runs But Zero Traces Appear in Console

Failure mode: FM-30 · Severity: high · Verified: 2026-06-22 (lab: langchain-langgraph) Stack: python · SDK 2.3.0 · integration: langchain/langgraph · model: llama3.2:3b (Ollama)

Symptom

The LangGraph app runs without errors, LLM calls succeed, and the user sees correct output — but the Galileo Console shows zero new traces for the session. No exception is raised. May present as "logs vanished again" or "Galileo stopped working after refactor."

Timing clue: if the run is notably faster than before (~1.8s vs ~4.0s for the same workload), the callback overhead is missing — Galileo is not wired in.

Quick check (under 2 minutes)

# Add this BEFORE invoking the graph; if it prints nothing, callback is absent
import logging
logging.basicConfig(level=logging.DEBUG)
# Look for "galileo" lines in the output

Or print the graph config/callbacks at runtime:

print("Callbacks on graph invoke:", config.get("callbacks"))
# Should show [<GalileoCallback ...>]; if [], callback is absent

Root cause

GalileoCallback is a LangChain BaseCallbackHandler. LangGraph (like all LangChain runtimes) does not auto-discover or auto-inject it. It must be explicitly passed to:

  1. The ChatOllama / ChatOpenAI / any LLM constructor (callbacks=[cb]), AND
  2. The graph.invoke(state, config={"callbacks": [cb]}) call.

If either is missing, the corresponding spans are silently dropped. The result graph output is still correct — the LLM is still called — but Galileo receives no data.

Common triggers: - A refactor that moves LLM instantiation inside a new helper function without forwarding the callback. - Adding a new node that creates its own LLM instance without accepting the shared callback. - Copying the working invoke() call but forgetting to pass config={"callbacks": callbacks}.

Fix

Step 1 — Create one shared GalileoLogger + GalileoCallback before building the graph

from galileo import GalileoLogger
from galileo.handlers.langchain import GalileoCallback

logger = GalileoLogger(project="galileo-labs", log_stream="langchain-langgraph")
cb = GalileoCallback(
    galileo_logger=logger,
    start_new_trace=True,      # start a fresh root trace for each chain invocation
    flush_on_chain_end=True,   # flush automatically when the top-level chain completes
)
callbacks = [cb]

Step 2 — Pass callbacks to every LLM constructor

llm = ChatOllama(model="llama3.2:3b", callbacks=callbacks)
# NOT: llm = ChatOllama(model="llama3.2:3b")  ← spans from this LLM won't be captured

Step 3 — Pass config={"callbacks": callbacks} to graph.invoke()

result = graph.invoke(
    {"question": question, ...},
    config={"callbacks": callbacks},   # ← required
)

Full minimal example

from galileo import GalileoLogger
from galileo.handlers.langchain import GalileoCallback
from langchain_ollama import ChatOllama
from langgraph.graph import StateGraph, END

logger = GalileoLogger(project="galileo-labs", log_stream="langchain-langgraph")
cb = GalileoCallback(galileo_logger=logger, start_new_trace=True, flush_on_chain_end=True)
callbacks = [cb]

llm = ChatOllama(model="llama3.2:3b", callbacks=callbacks)  # ← wire here

def my_node(state):
    return {"answer": llm.invoke([HumanMessage(content=state["question"])]).content}

builder = StateGraph({"question": str, "answer": str})
builder.add_node("respond", my_node)
builder.set_entry_point("respond")
builder.add_edge("respond", END)
graph = builder.compile()

result = graph.invoke(
    {"question": "What is gradient descent?", "answer": ""},
    config={"callbacks": callbacks},   # ← wire here too
)
# → 1 trace, 1 LLM span visible in Console

Verify

After applying the fix, re-run the query and check:

  1. Console: app.galileo.ai → galileo-labs → langchain-langgraph → New traces appear with 2+ LLM spans each (one per node that calls an LLM).
  2. Timing: The run takes noticeably longer than the no-callback version (~4s vs ~1.8s for a small Ollama model) due to the callback + flush overhead — this is expected and confirms the callback is active.
  3. No error on success: flush_on_chain_end=True handles flushing automatically; no manual logger.conclude() or logger.flush() needed for synchronous LangGraph runs.

Prevention

  • Add a guard assertion in your graph builder / test harness: python assert callbacks, "Galileo callbacks not set — traces will be lost"
  • Centralise LLM construction: one factory function that always accepts callbacks and raises if they're absent.
  • Integration test: after a run, verify trace count increased in the Console (use the Galileo API or MCP get_signals to count rows in the log stream).

Root cause

FM-31a: OpenAI API does not include usage stats in streaming chunks by default. The final chunk's usage field is None. The Galileo wrapper has nothing to capture → logs 0 tokens.

FM-31b: OpenAI SDK v2.x changed CompletionUsage from a plain object with .get() to a Pydantic model (no .get() method). The streaming finalization code in Galileo 2.3.0 (response_generator.py:_finalize) calls usage.get(...) directly — a pattern that worked with OpenAI v1.x but crashes on v2.x.

Root cause code (response_generator.py:175):

# BUG: usage is a Pydantic CompletionUsage (OpenAI v2.x) — no .get() method
num_input_tokens=usage.get("input_tokens", 0),
num_output_tokens=usage.get("output_tokens", 0),

The non-streaming path in extractors.py:_parse_usage() correctly uses usage.__dict__ and handles the field rename (prompt_tokens → input_tokens). The streaming _finalize() missed this normalization.

Key behavior to know: galileo.openai uses wrapt.wrap_function_wrapper for global monkey-patching. Even if you import openai separately (e.g., import openai as real_openai), any call through that client will go through Galileo's instrumentation — including the buggy streaming path. There is no "escape" via a separate client reference.

Fix

Option A — Recommended: Use stream=False for Galileo-instrumented calls

Token counts are captured correctly in the non-streaming path. For streaming UX, simulate token-by-token output from the completed response:

from galileo import galileo_context
from galileo.openai import openai as g_openai
import time

with galileo_context(project="my-project", log_stream="my-stream"):
    resp = g_openai.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
        stream=False,   # ← KEY: non-streaming captures tokens correctly
    )

answer = resp.choices[0].message.content
usage = resp.usage
# Tokens correctly captured: usage.input_tokens, usage.output_tokens

# If you need a streaming-like UI experience:
for word in answer.split():
    print(word, end=" ", flush=True)
    time.sleep(0.02)

Option B — Local SDK patch (if true streaming is required)

Apply this once at the top of your file. Patches _finalize() to normalize the usage object the same way the non-streaming _parse_usage() does:

# FM-31b local fix: normalize CompletionUsage before .get() calls
# Remove when galileo updates response_generator.py
from galileo.openai import response_generator as _rg

_orig_finalize = _rg.ResponseGeneratorSync._finalize

def _patched_finalize(self):
    # Normalize usage: convert Pydantic model to dict if needed
    if hasattr(self, '_usage') and self._usage is not None:
        u = self._usage
        if not isinstance(u, dict):
            d = u.__dict__.copy() if hasattr(u, '__dict__') else {}
            # Handle v1→v2 field rename (belt-and-suspenders)
            if 'completion_tokens' in d:
                d['output_tokens'] = d.pop('completion_tokens')
            if 'prompt_tokens' in d:
                d['input_tokens'] = d.pop('prompt_tokens')
            self._usage = d
    _orig_finalize(self)

_rg.ResponseGeneratorSync._finalize = _patched_finalize

async def run_agent(query: str, context: str) -> str:
    logger = init_galileo_and_agent_control()
    logger.start_trace(input=query)
    try:
        answer = await call_llm(query, context)
        logger.conclude(output=answer)
        return answer
    except agent_control.DenyException as e:

Bug: invoke_protect ERROR status silently bypasses LLM judge (2026-06-16)

Root cause: On the community/starter Galileo cluster tier, LLM-backed metrics (context_adherence, completeness, context_adherence_luna, etc.) are not available in the Protect API. invoke_protect returns ExecutionStatus.error with

    # BUG: resp.text is always set, even on ERROR; code returned here unconditionally

resp_status_str = str(getattr(resp, "status", "")).upper()
if "ERROR" not in resp_status_str:

# Else: fall through to LLM judge

1. Quality gate in CI/CD — block deploys that regress quality

# In your deployment pipeline:
# 1. Run a canary set (20 queries) against the new model/prompt
# 2. Score with Galileo
# 3. Block deploy if context_adherence < 0.75

def quality_gate(run_tag: str, threshold: float = 0.75) -> bool:
    """Return True = pass (deploy OK), False = block."""
    # Fetch mean context_adherence for the tagged run from Galileo API
    score = get_mean_metric(project="galileo-labs", tag=run_tag, metric="context_adherence")
    if score < threshold:
        print(f"❌ Quality gate FAILED: context_adherence={score:.2f} < {threshold}")
        return False
    print(f"✅ Quality gate passed: context_adherence={score:.2f}")
    return True

2. Enable Luna-2 100%-traffic scoring in production

Console → Log Streams → [your-stream] → Metrics → Add
Metric: context_adherence (Luna-2)
Scope: ALL traffic (not sampled)
Alert: if rolling 10-trace mean < 0.70 → notify Slack/PagerDuty

100% traffic scoring at Luna-2 cost is ~96% cheaper than GPT-4-grade judges. This is the only way to catch regressions in the first few minutes, not the first few days.
